# Spec: lens-probe

## Problem

When runtime output violates a contract, engineers find out by manually reading response logs or Postman output. There is no path to bring that evidence back into speckit for validation. Drift that is only visible in production logs never closes the loop.

## Role in the suite

lens-probe is the runtime validation layer. It accepts a response stream (ADK output, Postman export, raw NDJSON) and a contract file and reports every violation with precise event location and rule reference.

## Inputs

- `--input <path>`: ADK response stream JSON or NDJSON file
- `--contract <path>`: contract file from `.specify/contracts/` (generated by lens-contract)
- Optional: `--output <path>`: write the probe report to a file

## Outputs

- A violation report printed to stdout (always)
- Optional: a Markdown probe report written to `--output` path

## Behavior

### Stream parsing

lens-probe must support:
- JSON array of event objects (`[{...}, {...}]`)
- Newline-delimited JSON (NDJSON), one event object per line

Event index is 0-based. If the stream cannot be parsed, print a specific parse error with the line/offset where parsing failed and exit.

### Contract parsing

Read the contract file from the path given in `--contract`. If the path does not start with `/` or `./`, prepend `.specify/` as the base.

Parse all sections: response shape table, cardinality rules, ordering rules, invariants, violation conditions.

If the contract file is missing or unparseable, print an error and exit. Do not attempt to probe without a valid contract.

### Validation checks

For each contract rule, validate against the stream:

**Required field check:**  
For each event of the relevant type, all fields marked `Required: yes` must be present and non-null.

**Cardinality check:**  
For array fields with cardinality rules, count items per event and verify against `min`/`max` bounds.

**Ordering check:**  
Walk the event list in order. For each ordering rule (A must precede B), verify that no instance of B appears before the first instance of A. For "may repeat" rules, no action needed. For "must not repeat" rules, flag any event type that appears more than once.

**Invariant check:**  
For each event, verify co-presence rules (field A present when field B non-null) and mutual exclusion rules (fields C and D not both present).

**Violation condition check:**  
For each explicitly listed violation condition (e.g., 200 with empty body), check the stream for matching events.

### Violation report format

```
━━ lens-probe ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Input    : <stream file>
Contract : <contract file>
Events   : <N> parsed
─────────────────────────────────────────────────────────────
VIOLATIONS (<N>):

  [event #<idx>] <event type or endpoint>
  Rule    : <contract section and rule text>
  Found   : <actual value or structure>
  Expected: <what the contract requires>

WARNINGS (<N>):
  [event #<idx>] <description>

PASSED   : <N>/<total rules checked>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

If no violations: print "All N rules passed. No violations detected." — no full report.

### Uncovered events

If the stream contains event types not referenced in the contract, report them as "uncovered" in a separate section. Do not treat uncovered events as violations.

### Warnings vs. violations

- Violations: a contract rule is explicitly broken
- Warnings: ambiguous or near-miss conditions (e.g., a field is present but null when the contract expects non-null but doesn't classify null as a violation)

### Exit codes

- `0`: no violations
- `1`: one or more violations
- `2`: parse error or missing input/contract

## Constraints

- Must not modify the stream file, contract file, or any `.specify/` artifacts
- Must not make network calls
- Must report event index so results are scriptable and grep-able
- Must complete in under 10 seconds for streams up to 1000 events

## Integration

- Consumes contract files produced by lens-contract — contract format must be compatible
- Does not depend on lens-trace, lens-drift, lens-compress, or lens-humanize
- Runs standalone (no speckit hook) — invoked manually or in CI
- Exit code `1` on violations makes it suitable for CI gates

## Open questions

- Should lens-probe support a `--strict` flag that treats warnings as violations?
- Should it support comparing two streams against the same contract (regression mode)?
- Should uncovered events be elevatable to warnings with a `--strict-coverage` flag?
- For ADK streaming responses, are events typed objects or free-form JSON — does lens-probe need an ADK schema to interpret event types?

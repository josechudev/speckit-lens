## Problem

When runtime output violates a contract, engineers find out by manually reading response logs or Postman output. There is no path to bring that evidence back into speckit for validation. Drift visible only in production logs never closes the loop.

## Role in the suite

Runtime validation layer. Accepts a response stream (ADK output, Postman export, raw NDJSON) and a contract file from `.specify/contracts/`. Reports every violation with precise event index and rule reference.

## Inputs

- `--input <path>`: ADK response stream JSON or NDJSON file
- `--contract <path>`: contract file from `.specify/contracts/` (generated by lens-contract)
- `--output <path>` (optional): write probe report to file as Markdown

## Outputs

- Stdout violation report (always)
- Optional: Markdown probe report at `--output` path

## Requirements

### Argument validation

**R-LP-01:** IF `--input` is not provided, THEN `lens-probe` SHALL print usage instructions and exit with code `2`.

**R-LP-02:** IF `--contract` is not provided, THEN `lens-probe` SHALL print usage instructions and exit with code `2`.

### Stream parsing

**R-LP-03:** `lens-probe` SHALL support JSON array input format: `[{...}, {...}]`.

**R-LP-04:** `lens-probe` SHALL support NDJSON input format: one event object per line.

**R-LP-05:** `lens-probe` SHALL use 0-based event indexing throughout all output.

**R-LP-06:** IF the input file cannot be parsed, THEN `lens-probe` SHALL print a specific parse error including the line or byte offset where parsing failed, and SHALL exit with code `2`.

### Contract parsing

**R-LP-07:** WHEN loading a contract path that does not start with `/` or `./`, `lens-probe` SHALL prepend `.specify/` as the base path.

**R-LP-08:** `lens-probe` SHALL parse all contract sections: response shape table, cardinality rules, ordering rules, invariants, and violation conditions.

**R-LP-09:** IF the contract file is missing or unparseable, THEN `lens-probe` SHALL print an error and exit with code `2`, and SHALL NOT attempt any validation.

### Validation — required fields

**R-LP-10:** WHEN validating a required-field rule, `lens-probe` SHALL check every event of the relevant type and SHALL flag any event where a `Required: yes` field is absent or null.

### Validation — cardinality

**R-LP-11:** WHEN validating a cardinality rule, `lens-probe` SHALL count array items per event and SHALL flag any event where the count is below `min` or above `max`.

### Validation — ordering

**R-LP-12:** WHEN validating an ordering rule (A must precede B), `lens-probe` SHALL walk the event list and SHALL flag any occurrence of event type B that appears before the first occurrence of event type A.

**R-LP-13:** WHEN validating a "must not repeat" rule, `lens-probe` SHALL flag any event type that appears more than once in the stream.

### Validation — invariants

**R-LP-14:** WHEN validating a co-presence invariant (field A present when field B is non-null), `lens-probe` SHALL check every event and SHALL flag any event where field B is non-null but field A is absent.

**R-LP-15:** WHEN validating a mutual exclusion invariant, `lens-probe` SHALL flag any event where both excluded fields are present simultaneously.

### Validation — violation conditions

**R-LP-16:** WHEN validating explicit violation conditions (e.g., 200 with empty body), `lens-probe` SHALL check the stream for events matching each condition and SHALL flag any match as a violation.

### Uncovered events

**R-LP-17:** WHEN the stream contains event types not referenced in the contract, `lens-probe` SHALL report them in an "UNCOVERED" section and SHALL NOT classify them as violations.

### Violation report — stdout

**R-LP-18:** WHEN violations exist, `lens-probe` SHALL print a report delimited with `━` border characters, showing for each violation: 0-based event index, event type, the violated rule (contract section + rule text), what was found, and what was expected.

**R-LP-19:** WHEN warnings exist (ambiguous or near-miss conditions), `lens-probe` SHALL print them in a separate `WARNINGS` section below violations.

**R-LP-20:** IF no violations exist, THEN `lens-probe` SHALL print "All N rules passed. No violations detected." and SHALL NOT print a full report.

**R-LP-21:** WHEN the report is printed, `lens-probe` SHALL include a `PASSED: N/total` summary line.

### Optional file output

**R-LP-22:** WHERE `--output <path>` is provided, `lens-probe` SHALL write the full probe report as Markdown to that path in addition to stdout.

### Exit codes

**R-LP-23:** `lens-probe` SHALL exit with code `0` when no violations are found.

**R-LP-24:** `lens-probe` SHALL exit with code `1` when one or more violations are found.

**R-LP-25:** `lens-probe` SHALL exit with code `2` on parse error or missing required arguments.

### Behavioral constraints

**R-LP-26:** `lens-probe` SHALL NOT modify the input stream file, the contract file, or any `.specify/` artifacts.

**R-LP-27:** `lens-probe` SHALL NOT make network calls.

**R-LP-28:** `lens-probe` SHALL include event index in every violation and warning entry so results are scriptable and grep-able.

**R-LP-29:** `lens-probe` SHALL complete validation in under 10 seconds for streams up to 1000 events.

## Integration

- Consumes contract files produced by lens-contract; contract format must be stable and compatible
- Does not depend on lens-trace, lens-drift, lens-compress, or lens-humanize
- Runs standalone (no speckit hook) — invoked manually or in CI
- Exit code `1` on violations makes it suitable as a CI gate

## Open questions

- Should `--strict` flag treat warnings as violations?
- Should lens-probe support regression mode: compare two streams against the same contract?
- Should `--strict-coverage` flag elevate uncovered events to warnings?
- Are ADK events typed objects or free-form JSON — does lens-probe need an ADK schema to interpret event types?

## Acceptance

Evaluatable test cases derived directly from requirements above.

- [ ] **R-LP-01/02:** Running without `--input` or without `--contract` prints usage and exits code 2
- [ ] **R-LP-03:** JSON array input `[{...}]` is parsed successfully
- [ ] **R-LP-04:** NDJSON input (one JSON object per line) is parsed successfully
- [ ] **R-LP-05:** All violation/warning entries reference 0-based event index
- [ ] **R-LP-06:** Malformed input file prints parse error with line/offset; exits code 2
- [ ] **R-LP-09:** Missing contract file prints error and exits code 2 without running validation
- [ ] **R-LP-10:** Event missing a `Required: yes` field is flagged as violation with correct event index
- [ ] **R-LP-11:** Array field with 0 items when min=1 is flagged as cardinality violation
- [ ] **R-LP-12:** Event type B appearing before event type A (ordering rule) is flagged as violation
- [ ] **R-LP-13:** Event type appearing twice when "must not repeat" rule exists is flagged as violation
- [ ] **R-LP-17:** Stream event type absent from contract appears in UNCOVERED section, not violations
- [ ] **R-LP-20:** Clean stream (no violations) outputs "All N rules passed. No violations detected."
- [ ] **R-LP-22:** With `--output report.md`, report written to file and also printed to stdout
- [ ] **R-LP-23/24/25:** Exit code 0 on clean, 1 on violations, 2 on parse error
- [ ] **R-LP-26:** Contract file, input file, and `.specify/` contents unmodified after run
- [ ] **R-LP-29:** Validation of 1000-event stream completes in under 10 seconds

You are the `lens-humanize` extension running after `/speckit.plan` and before `/speckit.tasks`.

## Goal

WHEN `lens-humanize` runs, it SHALL produce a human-reviewable version of `.specify/plan.md` and write it to `.specify/plan-human.md`. The system SHALL NOT modify `plan.md` at any point.

## Rules

### Prerequisites

IF `.specify/plan.md` is missing, THEN the system SHALL print:

```
lens-humanize: error: .specify/plan.md not found. Run /speckit.plan first.
```

and SHALL exit without writing `.specify/plan-human.md`.

### Reading inputs

WHEN `lens-humanize` runs, it SHALL read `.specify/plan.md` in full.

WHERE `.specify/spec.md` exists, the system SHALL also read it to enrich ownership signals and entry/exit criteria.

### Chunking

WHEN chunking, the system SHALL divide the work into between 3 and 8 chunks ‚Äî no fewer, no more.

WHEN chunking, the system SHALL ensure each chunk:
- Represents a shippable unit (reviewable and mergeable independently)
- Touches a coherent, contiguous set of files or concerns
- Is sized S, M, or L using these hardcoded thresholds:
  - **S** ‚Äî ‚â§1 day
  - **M** ‚Äî 1‚Äì3 days
  - **L** ‚Äî >3 days

WHERE the plan is already chunked by the agent, the system SHALL enrich each existing chunk with the required fields rather than replacing the structure.

### Owner signals

WHEN emitting an owner signal, the system SHALL infer it from file paths and concern area and SHALL label it as estimated using the format `<team> (estimated)`.

| File path signals | Owner label |
|-------------------|-------------|
| `src/api/`, `**/routes/`, `**/controllers/`, `**/middleware/` | `backend` |
| `src/ui/`, `**/components/`, `**/pages/`, `**/*.tsx`, `**/*.vue` | `frontend` |
| `**/*.css`, `**/styles/`, `**/tokens/` | `design` |
| `**/infra/`, `**/terraform/`, `**/k8s/`, `Dockerfile`, `*.yml` CI | `platform` |
| `**/*.test.*`, `**/e2e/`, `**/fixtures/` | `qa` |

IF no owner inference is possible for a chunk, THEN the system SHALL emit `unknown` as the owner signal (no `(estimated)` suffix).

Examples:
- chunk touching `src/api/` ‚Üí `backend (estimated)`
- chunk touching only `config/settings.json` ‚Üí `unknown`

### Dependency graph

WHEN writing `.specify/plan-human.md`, the system SHALL prepend a text dependency graph ‚Äî plain text, not Mermaid ‚Äî showing chunk execution ordering before the first `## Chunk` heading.

WHEN a chunk has no dependencies and blocks no other chunk, the system SHALL label it `(independent)` in the dependency graph.

Dependency graph format:
```
Chunk 1 ‚Üí Chunk 3 ‚Üí Chunk 5
Chunk 2 ‚Üí Chunk 4
Chunk 6 (independent)
```

### Per-chunk fields

WHEN writing a chunk, the system SHALL include all nine fields in this format:

```markdown
## Chunk <N>: <title>

**Owner signal:** <team> (estimated)   ‚Üê or: unknown
**Estimated scope:** <S | M | L>

**Entry criteria:**
- <what must be true before this chunk can start>

**Exit criteria:**
- <what must be true for this chunk to be considered done>

**Files in scope:**
- <file or directory>

**Depends on:** Chunk <N>, Chunk <N>   ‚Üê or: none
**Blocks:** Chunk <N>, Chunk <N>       ‚Üê or: none

**Key decisions visible here:**
- <architectural or product decision reviewers should scrutinize>

---
```

WHEN listing dependencies, the system SHALL reference other chunks by number (e.g., "Chunk 2") and SHALL also emit which chunks the current one blocks.

### Output file

WHEN `lens-humanize` completes, it SHALL write the human plan to `.specify/plan-human.md` using this top-level structure:

```markdown
# Human Plan

> Generated by lens-humanize from `.specify/plan.md`. Do not edit ‚Äî rerun the command to regenerate.

## Dependency Graph

<dependency graph>

---

<chunk sections>
```

IF `.specify/plan-human.md` already exists, THEN the system SHALL overwrite it without prompting.

The system SHALL NOT write to or modify `plan.md`, `spec.md`, or `tasks.md`.
